# 🗺️ ChestX AI Platform — 项目文件详解

> 基于 YOLOv11 的目标检测智能体平台 | 前后端完整讲解 | 面向小白

---

## 一、这个项目是干什么的？

这是一个**"基于 YOLOv11 的目标检测智能体平台"**，通俗说就是：

> 用户上传图片 → AI 自动检测图片里的目标（飞机、汽车、人等）→ 还能用对话方式操作整个流程

整个项目分为两大块：

```
📦 fengxianghudong/
├── 🐍 backend/          ← 后端（Python）：处理数据、AI推理、存数据库
├── 🟢 frontend/         ← 前端（Vue.js）：用户看到的网页界面
├── 🐳 docker-compose.yml ← 一键启动数据库等基础设施
└── 📖 XJTU-fxhd/        ← 项目文档
```

## 二、比喻理解：把项目比作餐厅 🍽️

| 餐厅概念 | 对应项目 | 说明 |
|----------|----------|------|
| **菜单/服务员** | **前端 (frontend/)** | 用户看到的页面，点菜（发请求） |
| **厨房** | **后端 (backend/)** | 处理订单、做菜（业务逻辑） |

| **菜谱** | **AI 模型 (YOLOv11)** | 做检测的核心算法 |

## 后端文件逐个讲

### 1. `backend/main.py` — 后端大门 🚪

> 作用：整个后端的入口文件，程序从这里启动  
> 类比：餐厅的大门，所有请求都先经过这里

**它做了这些事：**
- 创建 FastAPI 应用实例（第 32 行 `app = FastAPI(...)`）
- 配置 CORS（允许前端跨域访问，第 52 行）
- 注册路由（把 `/api/auth`、`/api/health` 等接口挂载上来，第 62-63 行）
- 注册中间件（请求日志记录，第 58 行）
- 注册异常处理器（出错时返回统一格式，第 48 行）
- 启动时初始化 MinIO 存储桶

**关键概念 — CORS（跨域资源共享）**：
前端在 `localhost:5173`，后端在 `localhost:8000`，浏览器默认不允许不同端口之间通信。CORS 就是告诉浏览器"允许前端来访问我"。

**关键概念 — 中间件（Middleware）**：
相当于一个"安检通道"，每个请求到达路由之前，先经过中间件处理。比如记录日志、检查登录状态。

**文件关系链**：
```
main.py 依赖:
  ├── app/api/auth.py        ← 认证接口（登录/注册）
  ├── app/api/health.py      ← 健康检查接口
  ├── app/config/settings.py ← 全局配置
  ├── app/core/exceptions.py ← 异常处理
  ├── app/middleware/        ← 请求日志中间件
  └── app/storage/           ← MinIO 初始化
```

### 4. `backend/app/database/session.py` — 数据库大门 🗄️

> 作用：管理数据库连接，提供"拿连接→用完归还"的机制  
> 类比：餐厅仓库管理员，你需要东西时找他拿，用完他收回去

**核心概念 — 连接池（Connection Pool）：**
- 每次请求不需要重新建立数据库连接（很慢）
- 而是从"池子"里拿一个现成的连接，用完放回去
- `pool_size=10` 表示池子里保持 10 个连接

**核心概念 — `get_db()` 依赖注入：**
```python
# 这个函数是 FastAPI 的"依赖注入"机制
# 其他路由只需要写 Depends(get_db)，就能自动获得数据库连接
# FastAPI 会自动在请求结束后关闭连接（finally: db.close()）
```

**重要概念 — `Base = declarative_base()`：**
所有数据库表的模型类（User、DetectionTask 等）都继承自这个 `Base`。它就像一个"登记处"，所有继承它的类都会被自动注册。

**文件关系**：
```
session.py 被以下文件引用:
  ├── app/entity/db_models.py  ← 模型类继承 Base
  ├── app/api/auth.py          ← 路由中用 get_db
  ├── app/services/            ← 服务层用 Session
  └── alembic/env.py           ← 数据库迁移用 Base.metadata
```

### 5. `backend/app/entity/db_models.py` — 数据表定义 📊

> 作用：用 Python 类定义数据库中的每一张表  
> 类比：设计仓库货架的规格，每个货架放什么类型的东西

**核心概念 — ORM（对象关系映射）：**
- 以前：写 SQL 语句 `SELECT * FROM users WHERE id=1`
- 现在：写 Python `db.query(User).filter(User.id == 1).first()`
- ORM 自动把 Python 代码翻译成 SQL

**这个文件定义了以下表（目前是骨架，部分表后续完善）：**

**用户权限体系：**
- `User` (用户表) — 存用户名、密码（加密）、邮箱
- `Role` (角色表) — 管理员/操作员/访客
- `Permission` (权限表) — 细粒度权限
- `UserRole` (用户-角色关联) — 一个用户可以有多个角色
- `RolePermission` (角色-权限关联)

**检测业务：**
- `DetectionScene` (检测场景) — 不同业务方向（遥感/工业/农业）
- `DetectionTask` (检测任务) — 每次检测操作记录
- `DetectionResult` (检测结果) — 每张图片检测出的目标

**模型管理：**
- `TrainingTask` (训练任务) — 模型训练记录
- `ModelVersion` (模型版本) — 训练出的模型文件管理

**智能体：**
- `ChatSession` (对话会话) — 一次对话的上下文
- `ChatMessage` (对话消息) — 具体每条消息

**系统运维：**
- `OperationLog` (操作日志) — 记录用户操作

**表之间的关系（外键 ForeignKey）：**
```python
# 例如：每条检测任务属于某个用户和某个场景
class DetectionTask(Base):
    user_id = Column(ForeignKey("users.id"))            # 外键 → 指向 users 表
    scene_id = Column(ForeignKey("detection_scenes.id")) # 外键 → 指向 detection_scenes 表
```

### 6. `backend/app/entity/schemas.py` — 数据校验模板 📋

> 作用：定义 API 接口"收什么格式的数据"和"返回什么格式的数据"  
> 类比：餐厅的"标准订单格式"，顾客必须按格式填写

**核心概念 — Pydantic vs SQLAlchemy：**

| | SQLAlchemy Model (db_models.py) | Pydantic Schema (schemas.py) |
|---|---|---|
| 用途 | 和数据库交互 | 和前端交互 |
| 所在环节 | 后端内部 | API 边界 |
| 例子 | `User` 类有 `hashed_password` 字段 | `UserResponse` 类**没有**密码字段 |

**为什么要分两层？**
安全！数据库里存了加密密码，但返回给前端时绝对不能包含密码字段。Pydantic Schema 就负责"过滤"。

**这个文件里的模型分类：**
- `*Create`（如 `UserRegister`）：前端发来的创建请求
- `*Response`（如 `UserResponse`）：后端返回给前端的数据
- `TokenResponse`：登录成功返回的 JWT Token + 用户信息

### 7. `backend/app/api/auth.py` — 认证接口 🔐

> 作用：提供登录、注册、获取用户信息的 API 接口  
> 路由前缀：`/api/auth`

**提供的接口：**

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/auth/register` | POST | 用户注册 |
| `/api/auth/login` | POST | 用户登录，返回 JWT Token |
| `/api/auth/me` | GET | 获取当前登录用户信息（需要 Token）|

**核心概念 — JWT Token 认证流程：**
```
1. 用户登录 → 后端验证用户名密码 → 生成 JWT Token → 返回给前端
2. 前端存 Token 到 localStorage → 后续请求在 Header 带 Authorization: Bearer <token>
3. 后端从 Token 解析出用户 ID → 查数据库获取用户信息
```

**核心概念 — `Depends(get_current_user)`：**
这是一个"依赖项"，用在需要登录的路由中。FastAPI 会自动：
1. 从请求头提取 Token
2. 解析 Token 获取用户 ID
3. 从数据库查出用户 → 注入到路由函数

> 作用：检查各服务是否正常（数据库、Redis、MinIO）  
> 用途：Docker 探活、运维监控

两个接口：
- `GET /api/health`：轻量检查（只看应用是否活着）

### 9. `backend/app/core/security.py` — 安全模块 🔒

> 作用：密码加密 + JWT Token 生成/解析

**四个核心函数：**

| 函数 | 作用 |
|------|------|
| `hash_password()` | 明文密码 → bcrypt 哈希（不可逆） |
| `verify_password()` | 检查用户输入的密码是否匹配 |
| `create_access_token()` | 生成 JWT Token（含用户ID + 过期时间） |
| `decode_access_token()` | 解析 JWT Token，拿出里面的数据 |

**为什么密码不能明文存储？**
如果数据库泄露，攻击者也拿不到真实密码（bcrypt 不可逆）。

### 10. `backend/app/core/exceptions.py` — 异常处理 🚨

> 作用：捕获所有错误，返回统一格式的 JSON 响应

**统一响应格式：**

### 15. `backend/alembic/` — 数据库迁移工具 🔄

> 作用：管理数据库表结构的版本，像 Git 管理代码一样  
> 类比：仓库改造的记录本，记录每次货架布局的变化

**工作流程：**
```
1. 修改 db_models.py（加了新表或新字段）
2. 执行 alembic revision --autogenerate -m "添加了xxx表"
3. Alembic 自动对比模型和实际数据库，生成迁移脚本
4. 执行 alembic upgrade head → 把改动应用到数据库
```

`alembic/env.py` 是迁移的配置文件，核心是这行：
```python
target_metadata = Base.metadata  # 告诉 Alembic 从哪里读取表结构
```

### 2. `frontend/vite.config.js` — 前端控制台 ⚙️

> 作用：Vite 的配置中心

**关键配置：**

**① 路径别名 `@`：**
```js
alias: { "@": path.resolve(__dirname, "src") }
// 之后写 import xxx from "@/utils/request" 就等于 import xxx from "src/utils/request"
```

**② 开发服务器代理（最重要的概念！）：**
```js
proxy: {
  "/api": {
    target: "http://localhost:8000",  // 所有 /api 开头的请求转发到后端
    changeOrigin: true,
  },
}
```

**为什么需要代理？**
- 前端跑在 `localhost:5173`
- 后端跑在 `localhost:8000`
- 前端代码发请求 `axios.get("/api/auth/me")` 实际请求的是 `localhost:5173/api/auth/me`
- Vite 代理自动把这个请求转发到 `localhost:8000/api/auth/me`
- **规避了浏览器的跨域限制**

### 6. `frontend/src/router/index.js` — 路由（页面导航）🗺️

> 作用：定义 URL 和页面组件的对应关系，控制页面跳转  
> 类比：餐厅的菜单，顾客说"我要去xxx"，服务员带他去对应桌子

**路由表：**

| URL 路径 | 对应组件 | 需要登录？ |
|----------|----------|-----------|
| `/login` | LoginPage.vue | ❌ 不需要 |
| `/register` | RegisterPage.vue | ❌ 不需要 |
| `/chat` | ChatPage.vue | ✅ 需要 |
| `/detection` | DetectionPage.vue | ✅ 需要 |
| `/training` | TrainingPage.vue | ✅ 需要 |
| `/history` | HistoryPage.vue | ✅ 需要 |
| `/dashboard` | DashboardPage.vue | ✅ 需要 |

**核心概念 — 路由守卫（Navigation Guard）：**
```js
router.beforeEach((to, from, next) => {
  // 每次页面跳转前都会执行这个函数
  // 检查：如果目标页面需要登录但用户没登录 → 强制跳到登录页
})
```

**核心概念 — 懒加载：**
```js
component: () => import("@/views/ChatPage.vue")
// 不是一次性加载所有页面，而是访问时才加载 → 首屏更快
```

**核心概念 — 嵌套路由：**
```
/ (根路径)
  └── MainLayout.vue (外壳：顶部导航 + 侧边栏)
        ├── /chat       → ChatPage.vue
        ├── /detection  → DetectionPage.vue
        ├── /training   → TrainingPage.vue
        └── ...
```
`MainLayout.vue` 是"布局外壳"，页面内容通过 `<router-view />` 渲染在中间区域。

### 7. `frontend/src/stores/` — 全局状态管理 🗃️

> 作用：管理跨页面共享的数据（如登录状态、用户信息）

**核心概念 — 为什么需要 Pinia？**
- 用户登录后，多个页面都需要知道"当前用户是谁"
- 不能每个页面都去请求一遍后端 → 存在全局 Store 里，所有页面共享

**`stores/index.js`** — Pinia 入口，创建实例。

**`stores/user.js`** — 用户状态：

| 状态/方法 | 说明 |
|-----------|------|
| `token` | JWT Token（存在 localStorage 中，刷新不丢失）|
| `user` | 用户信息（用户名、邮箱、角色等）|
| `isLoggedIn` | 计算属性：是否已登录 |
| `login()` | 登录操作：调接口 → 存 Token → 存用户信息 |
| `logout()` | 退出：清空 Token 和用户信息 |
| `fetchUserInfo()` | 刷新用户信息 |

**`stores/agent.js`** — 智能体对话状态：

| 状态/方法 | 说明 |
|-----------|------|
| `messages` | 当前对话的消息列表 |
| `isLoading` | 是否正在等待 AI 回复 |
| `addMessage()` | 添加一条消息 |
| `abort()` | 中断 AI 回复 |

### 9. `frontend/src/utils/request.js` — Axios 封装 📡

> 作用：统一配置 Axios（baseURL、拦截器、错误处理）

**核心概念 — 拦截器（Interceptor）：**

**请求拦截器**（发请求前自动做的事）：
```js
// 自动从 store 拿 Token，加到请求头
config.headers.Authorization = `Bearer ${userStore.token}`
```

**响应拦截器**（收到响应后自动做的事）：
```js
// 401 → Token 过期，强制退出登录
// 403 → 权限不足，提示用户
// 500 → 服务器错误
```

**数据流：**
```
页面组件 → api/auth.js → request.js (axios) → [Vite 代理] → 后端
```

### 10. `frontend/src/utils/stream.js` — SSE 流式处理 🌊

> 作用：处理 AI 对话的"逐字输出"效果（像 ChatGPT 那样）

**核心概念 — SSE (Server-Sent Events)：**
- 普通请求：客户端发请求 → 服务端返回 → 结束
- SSE：客户端发请求 → 服务端持续推送数据 → 直到结束
- 实现效果：AI 回复一个字一个字地出现在屏幕上
