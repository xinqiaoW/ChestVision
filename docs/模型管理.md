# 模型管理

## 目标

`模型管理`负责统一管理系统中可用于 detection 推理的模型。模型来源分为两类：

- 训练得到的模型：训练任务完成后自动登记。
- 用户上传的模型：管理员通过模型管理页面上传。

模型文件的权威存储位置是 OSS。后端数据库保存模型元数据、来源关系、OSS 对象位置和默认推理模型配置；服务端本地文件只作为推理缓存使用。

## 核心决策

### 模型主表

每个可管理模型都必须有唯一模型 ID。

采用 `model_versions` 作为所有模型的主表：

- 训练得到的模型：`training_task_id != null`
- 用户上传的模型：`training_task_id == null`
- 每条 `model_versions` 记录代表一个可管理模型版本
- 页面、接口、删除、下载、设置默认都以 `model_version_id` 为主标识

上传过程不直接写入最终模型主表。建议新增 `model_uploads` 记录上传会话：

- 上传未完成时，只存在于 `model_uploads`
- 上传完成且 OSS 对象校验通过后，才创建 `model_versions`
- `model_uploads.model_version_id` 指向最终模型版本

这样可以避免半上传模型污染“所有模型”列表。

### 模型状态

`model_versions.status = active` 表示模型当前可被系统使用和展示：

- OSS 权重文件存在
- 数据库元数据完整
- 未被删除
- 可下载
- 可配置为某场景默认推理模型

`active` 不表示当前默认模型。默认模型由 `is_default = true` 表示。

### 默认推理模型

默认推理模型按场景生效。

当前系统只有一个主要场景，因此页面上看起来像全局唯一默认模型；但后端仍按 `scene_id` 设计：

- 同一 `scene_id` 下最多一个 `is_default = true` 的模型
- 设置默认模型时只影响同一场景
- 后续新增场景时不需要重新定义默认模型语义

### 模型格式

第一阶段暂不支持 ONNX。

仅支持 YOLO `.pt` 权重，模型类型包括：

- `yolo11n`
- `yolo11s`
- `yolo11m`
- `yolo11l`
- `yolo11x`

后端和前端都必须校验：

- `model_type` 属于上述枚举
- 上传文件后缀为 `.pt`

## OSS 存储设计

### 训练模型

训练完成后自动登记训练产物中的权重文件。

远程训练输出权重示例：

```text
{REMOTE_TRAIN_OSS_PREFIX}/train/jobs/{task_uuid}/weights/best.pt
```

训练完成后，后端应：

1. 确认 `best.pt` 在 OSS 中存在。
2. 创建或更新 `model_versions`。
3. 创建或更新 `model_artifact_locations`：
   - `artifact_type = best_weight`
   - `storage_backend = oss`
   - `bucket = OSS_BUCKET`
   - `object_key = best.pt object key`
   - `is_primary = true`
4. 写入文件大小、ETag 等元数据。

训练模型不需要用户手动“导出”后才进入模型管理。

### 上传模型

用户上传模型使用独立 OSS 前缀，例如：

```text
{REMOTE_TRAIN_OSS_PREFIX}/models/uploaded/{user_id}/{upload_id}/model.pt
```

上传采用统一的 OSS 文件分片上传模块：

1. 前端请求创建上传会话。
2. 后端创建 `model_uploads`，通过 OSS 文件分片上传模块初始化 multipart。
3. 前端分批请求 part URL。
4. 浏览器逐片 PUT 到 OSS。
5. 前端提交全部 `part_number + etag`。
6. 后端调用 OSS CompleteMultipartUpload。
7. 后端 HeadObject 校验对象。
8. 后端把 `model_uploads.status` 写为 `UPLOADED`。
9. 后端创建 `model_versions` 和 `model_artifact_locations`。

上传完成后模型进入 `active`，但不会自动替换当前默认推理模型。

## 下载设计

模型权重下载应面向大文件设计。

第一优先级是返回短期 OSS GET 预签名 URL，让浏览器直接从 OSS 下载，后端不代理大文件内容。

如果 OSS 和浏览器环境支持 Range 请求，应规划分片/断点下载能力：

- 后端签发可下载 URL
- OSS Bucket CORS 允许 `GET` 和 `Range`
- OSS 暴露必要响应头，例如 `Content-Length / Content-Range / ETag`
- 前端可以按 Range 分片下载并展示下载进度

同样的下载策略也应扩展到 `数据集管理`页面，用于下载数据集 ZIP。

## 删除设计

当前开发阶段采用联动强删除策略。

### 删除模型

删除模型时，如果该模型来源于训练任务，则存在模型和训练记录互相引用关系。

删除模型流程：

1. 前端点击模型 `删除`。
2. 后端检查该模型是否关联 `training_task_id`。
3. 如果关联训练任务，前端必须弹窗提示：
   - 该模型来源于训练任务
   - 删除模型会同时删除源训练记录
   - 删除会移除数据库记录和 OSS 对象
4. 用户确认后执行联动删除：
   - 删除模型 OSS 权重和关联 artifact
   - 删除 `model_artifact_locations`
   - 删除 `model_versions`
   - 删除关联训练任务产生的 OSS 输出
   - 删除训练任务相关数据库记录

### 删除训练记录

删除训练记录时，如果该训练已经登记模型，也必须联动删除模型。

删除训练记录流程：

1. 前端点击训练任务 `删除`。
2. 后端检查是否存在 `model_versions.training_task_id = task.id`。
3. 如果存在模型，前端必须弹窗提示：
   - 该训练任务已生成模型
   - 删除训练任务会同时删除模型
   - 删除会移除数据库记录和 OSS 对象
4. 用户确认后执行联动删除。

### 默认模型删除限制

如果要删除的模型是当前默认推理模型，应先要求用户选择新的默认模型，或在同一个确认流程中明确替换默认模型后再删除。

不允许删除后留下场景没有可用默认模型，除非系统明确允许“无默认模型”状态并能在检测接口中给出清晰错误。

## 设置默认推理模型

设置默认推理模型时，要求同步 OSS 权重到服务端本地缓存，并执行 `reload_model()`。

关键顺序必须是：

1. 从 OSS 下载新模型到本地临时缓存文件。
2. 下载成功并完成基础校验后，替换本地默认缓存文件。
3. 更新数据库默认模型状态。
4. 调用 detection 服务 `reload_model()`。
5. 删除旧的本地默认缓存文件。

注意：第 5 步只删除旧的本地默认缓存文件，不删除旧模型版本记录，也不删除旧模型的 OSS 权重文件。

旧模型仍应保留在“所有模型”列表中，除非用户显式删除它。

如果任一步失败：

- 不能进入下一步
- 数据库默认模型不应被提前切换
- 页面需要展示同步失败原因

## 本地缓存状态

`模型管理`页面可以展示默认模型同步状态。

建议后端返回：

- `cache_status`: `synced / syncing / failed / unknown`
- `cache_path`
- `cache_updated_at`
- `cache_error`

第一阶段也可以不新增字段，只在设置默认模型接口返回同步结果；但页面上应能表达“默认推理模型已同步到本地缓存”或“同步失败”。

## 后端 API 建议

### 查询所有模型

```text
GET /api/model-management/models
```

筛选参数：

- `model_name`
- `version`
- `scene_id`
- `model_type`
- `source_type`: `trained / uploaded / all`

返回模型列表，以 `model_version_id` 为主标识。

### 查询当前默认推理模型

```text
GET /api/model-management/default-model?scene_id={scene_id}
```

返回同一场景下 `is_default = true` 的模型，以及本地缓存同步状态。

### 创建模型上传会话

```text
POST /api/model-management/uploads
```

创建 OSS multipart 上传会话，只接受 YOLO `.pt` 模型。

### 签发模型分片上传 URL

```text
POST /api/model-management/uploads/{upload_id}/multipart/parts/sign
```

### 完成模型上传

```text
POST /api/model-management/uploads/{upload_id}/multipart/complete
```

完成后创建 `model_versions` 和 `model_artifact_locations`。

### 取消模型上传

```text
POST /api/model-management/uploads/{upload_id}/multipart/abort
```

### 删除模型

```text
DELETE /api/model-management/models/{model_version_id}
```

如果存在联动删除关系，前端必须确认后再调用。接口可要求显式参数：

```text
cascade=true
```

### 设置默认推理模型

```text
POST /api/model-management/models/{model_version_id}/set-default
```

接口必须完成 OSS 到本地缓存同步，并在成功后更新数据库和 reload 模型。

### 下载模型权重

```text
GET /api/model-management/models/{model_version_id}/download-url
```

返回短期 OSS 下载 URL。后续可扩展分片下载计划。

## 旧接口处理

不保留旧模型管理接口兼容。

后续实现新 `模型管理`页面后，可以移除旧接口：

```text
GET /api/training/models
POST /api/training/models/{model_version_id}/set-default
```

历史实现由 git 提交记录保留。

## 与前端页面的关系

`ModelPage.vue` 使用新的模型管理 API。

页面结构：

- 上半部分显示当前默认推理模型。
- 中间提供上传模型功能。
- 下半部分管理所有模型。

训练来源模型在列表中显示 `task_uuid`，点击后打开新页面进入：

```text
/training?task_uuid={task_uuid}
```
