# 模型管理页面

## 目标

`模型管理`页面用于统一管理两类模型：

- 训练得到的模型：远程训练或本地训练完成后生成的 `best.pt` 等权重文件。
- 用户上传的模型：管理员手动上传的 `.pt / .onnx` 等模型文件。

模型文件的权威存储位置是 OSS。后端只保存模型版本元数据、OSS object key、上传会话、状态和检测场景关联关系，不把用户上传的模型文件长期保存在后端本地目录中。

## 核心约束

- OSS 是所有模型文件的管理中心。
- 浏览器不接触 AccessKey 或 Secret，只拿后端签发的短期 OSS multipart part URL。
- 后端不得把用户上传模型持久化保存到 `backend/models` 作为权威数据源。
- `backend/models/best.pt` 只能作为推理缓存或兼容旧检测逻辑的本地副本，不能作为模型管理页面的数据源。
- 模型列表来自数据库 `model_versions` 和 `model_artifact_locations`，而不是扫描本地目录。
- 训练得到的模型由训练产物确认流程登记；用户上传的模型由上传完成确认流程登记。
- 删除模型应先更新数据库状态，再按策略删除或标记 OSS 对象；不能只删除本地文件。
- 默认模型只允许指向 `active` 状态且 OSS 主权重存在的模型版本。
- `模型训练`页面不再承担“设置 detection 默认模型”的职责；该能力迁移到 `模型管理`页面。
- `模型管理`页面展示当前默认推理模型，但不展示训练监控指标，也不展示“已加载/待加载/加载失败”等运行状态，避免和训练监控页、后端运行状态探测产生冗余。

## 数据模型设计

### 现有表复用

`model_versions` 继续表示业务模型版本：

- `training_task_id != null`：训练得到的模型。
- `training_task_id == null`：用户上传的模型。
- `status`：建议继续使用 `active / archived / deleted`。
- `is_default`：同一检测场景下只能有一个默认模型。
- `model_path`：保留给旧推理链路或本地缓存，不作为权威位置。
- `file_size / map50 / map50_95 / precision / recall`：用于列表展示。

`model_artifact_locations` 表示模型文件和相关报告的真实存储位置：

- `artifact_type = best_weight`：主模型权重。
- `artifact_type = eval_report / results_csv / success`：训练报告或辅助产物。
- `storage_backend = oss`：OSS 权威副本。
- `bucket + object_key`：OSS 对象定位。
- `is_primary = true`：当前模型版本的主权重。
- `lifecycle_state = active / deleted / cache / expired`：对象生命周期状态。

### 建议新增表

建议新增 `model_uploads`，用于管理用户上传模型的 multipart 会话，避免半上传文件直接污染 `model_versions`：

- `id`
- `upload_uuid`
- `user_id`
- `scene_id`
- `model_name`
- `version`
- `model_type`
- `filename`
- `content_type`
- `expected_size`
- `actual_size`
- `bucket`
- `object_key`
- `status`: `INITIATED / UPLOADING / UPLOADED / FAILED / EXPIRED / CANCELLED`
- `oss_upload_id`
- `part_size`
- `etag`
- `error_message`
- `model_version_id`
- `expires_at`
- `created_at`
- `updated_at`
- `completed_at`

如果当前开发阶段不想新增表，也可以临时把上传中状态写入 `model_versions.status = uploading`，但不推荐。上传会话和业务版本语义不同，分表更清楚。

## OSS Object Key 设计

训练得到的模型可以直接登记训练输出中的现有对象，不强制复制：

```text
{REMOTE_TRAIN_OSS_PREFIX}/train/jobs/{task_uuid}/weights/best.pt
```

用户上传模型建议使用独立前缀：

```text
{REMOTE_TRAIN_OSS_PREFIX}/models/uploaded/{user_id}/{upload_id}/model.pt
```

如果后续需要统一“模型中心”前缀，也可以在训练结束后把 `best.pt` 复制到：

```text
{REMOTE_TRAIN_OSS_PREFIX}/models/trained/{task_uuid}/best.pt
```

第一阶段建议不复制训练产物，直接登记原始训练输出 Object Key，减少 OSS 存储冗余。

## 后端 API 设计

建议新增独立命名空间，避免继续扩张旧的 `/api/training/models`：

```text
/api/model-management
```

### 查询模型列表

```text
GET /api/model-management/models
```

查询参数：

- `source_type`: `trained / uploaded / all`
- `status`: 默认 `active`
- `model_name`: 文本筛选，按模型名称模糊匹配
- `version`: 文本筛选，按版本模糊匹配
- `scene_id`: 下拉单选筛选；页面显示值来自 `DetectionScene.display_name`
- `model_type`: 下拉单选筛选，例如 `yolo11n / yolo11s / yolo11m / yolo11l / yolo11x / onnx`

返回字段：

- `id`
- `source_type`
- `model_name`
- `version`
- `model_type`
- `scene_id`
- `scene_name`
- `scene_display_name`
- `status`
- `is_default`
- `file_size`
- `training_task_id`
- `training_task_uuid`
- `created_at`
- `oss`: `bucket / object_key / uri / etag`

说明：后端可以继续返回 `map50 / map50_95 / precision / recall` 等指标供其他页面使用，但 `模型管理`页面默认不展示这些指标，训练指标应留在训练监控页中查看。

### 查询当前默认推理模型

```text
GET /api/model-management/default-model
```

返回当前 detection 默认模型。默认模型是 `model_versions.is_default = true` 的 active 模型，并且应存在 OSS 主权重。

返回字段：

- `id`
- `model_name`
- `version`
- `model_type`
- `scene_id`
- `scene_display_name`
- `source_type`: `trained / uploaded`
- `training_task_id`
- `training_task_uuid`
- `file_size`
- `created_at`
- `oss`: `bucket / object_key / uri / etag`

不返回或不展示训练监控指标；不返回或不展示运行时加载状态。页面只表达“当前默认推理模型是哪一个”。

### 创建用户上传会话

```text
POST /api/model-management/uploads
```

请求字段：

- `scene_id`
- `model_name`
- `version`
- `model_type`
- `filename`
- `content_type`
- `expected_size`
- `part_size`

后端行为：

1. 按 `model_type` 校验文件后缀。
2. 创建 `model_uploads` 记录。
3. 生成 OSS object key。
4. 调用 OSS `InitMultipartUpload`。
5. 返回 `upload_id`、`part_size`、签名接口、合并接口、取消接口。

模型类型与后缀第一阶段约束：

- `yolo11n / yolo11s / yolo11m / yolo11l / yolo11x`：只允许 `.pt`。
- `onnx`：只允许 `.onnx`。

后续如果支持 TensorRT 或其他推理格式，再扩展：

- `tensorrt`：`.engine`
- OpenVINO：可能需要 `.xml + .bin`，这不是单文件模型，需要单独设计模型包上传。

### 签发上传分片

```text
POST /api/model-management/uploads/{upload_id}/multipart/parts/sign
```

请求字段：

- `part_numbers`
- `expires_seconds`

返回每个分片的短期 PUT URL。前端逐片 PUT 到 OSS，并保存 OSS 返回的 `ETag`。

### 完成用户上传

```text
POST /api/model-management/uploads/{upload_id}/multipart/complete
```

请求字段：

- `parts`: `part_number + etag`

后端行为：

1. 调用 OSS `CompleteMultipartUpload`。
2. 调用 `HeadObject` 校验对象存在、大小和 ETag。
3. 创建 `model_versions`。
4. 创建 `model_artifact_locations`，其中 `artifact_type = best_weight`、`storage_backend = oss`、`is_primary = true`。
5. 把 `model_uploads.status` 更新为 `UPLOADED`，并写入 `model_version_id`。

### 取消用户上传

```text
POST /api/model-management/uploads/{upload_id}/multipart/abort
```

后端行为：

- 调用 OSS `AbortMultipartUpload`。
- 把 `model_uploads.status` 更新为 `CANCELLED`。
- 不创建 `model_versions`。

### 删除模型版本

```text
DELETE /api/model-management/models/{model_version_id}
```

删除策略：

1. 如果模型是默认模型，先拒绝删除，要求用户切换默认模型后再删。
2. 如果模型已有历史检测任务引用，默认软删除：`model_versions.status = deleted`，`model_artifact_locations.lifecycle_state = deleted`。
3. OSS 对象删除建议第一阶段同步执行 `DeleteObject`；如果担心历史追溯，可改为只标记并交给生命周期策略清理。
4. 本地缓存文件如果存在，可以删除，但不能作为主要删除依据。

### 设置默认模型

```text
POST /api/model-management/models/{model_version_id}/set-default
```

后端行为：

1. 校验模型状态为 `active`。
2. 校验存在 `best_weight` OSS 主对象。
3. 清除同场景其他模型默认标记。
4. 设置当前模型为默认。
5. 如果检测服务仍依赖本地文件，按需把 OSS 权重下载到本地推理缓存并热重载。

该接口由 `模型管理`页面的“配置”弹窗触发。`配置`弹窗第一阶段只做一件事：确认是否将所选模型替换为当前 detection 默认推理模型。

### 获取下载地址

```text
GET /api/model-management/models/{model_version_id}/download-url
```

返回短期 OSS GET 预签名 URL。不要把永久公开 URL 写入数据库。

## 训练产物登记流程

远程训练成功后，当前流程已经会确认 OSS 中的 `best.pt / results.csv / _SUCCESS` 等产物。模型管理需要补齐下面的版本登记：

1. 训练任务完成并确认 OSS 关键产物存在。
2. 后端创建或更新 `model_versions`：
   - `training_task_id = task.id`
   - `model_name = task.model_name`
   - `model_type = task.model_name`
   - `version` 可用 `task_uuid` 或递增版本号。
   - `status = active`
   - `file_size = best_weight HeadObject.content_length`
3. 将 `model_artifact_locations.best_weight` 关联到 `model_version_id`。
4. 将训练指标同步到 `model_versions.map50 / map50_95 / precision / recall`。

这样模型管理页可以无差别展示训练得到的模型和用户上传的模型。

## 前端页面设计

路由建议：

```text
/models
```

侧边栏标题：

```text
模型管理
```

页面主体采用管理台布局，不做营销式页面。主要区域：

- 上半部分：当前默认推理模型。
- 中间部分：上传模型入口。
- 下半部分：所有模型管理列表。

用户上传交互应复用数据集上传页的 multipart 控制逻辑：暂停中断当前 XHR，继续时重新签发当前 part URL，取消时中断 XHR 并调用后端 abort。

### 当前默认推理模型区域

上半部分用于展示当前 detection 默认推理模型，只回答“当前推理默认模型是哪一个”，不展示训练页已有的监控指标，不展示运行时加载状态。

建议展示字段：

- 模型名称：`model_name`
- 版本：`version`
- 场景类型：显示 `DetectionScene.display_name`
- 模型类型：`model_type`，例如 `yolo11n / yolo11m / onnx`
- 来源：
  - 训练来源：显示训练 ID，并可点击打开源训练任务监控页。
  - 上传来源：显示 `上传`。
- 权重文件大小：`file_size`
- 创建日期：`created_at`
- OSS 位置：`oss://{bucket}/{object_key}`，可以作为次要信息展示，长路径需要截断并在 hover 时显示完整值。

不展示字段：

- `mAP50 / mAP50-95 / Precision / Recall` 等训练指标。
- `已加载 / 待加载 / 加载失败` 等运行时状态。
- 训练监控页中已有的训练过程信息。

### 上传模型区域

上传模型功能位于默认推理模型区域和所有模型列表之间。第一阶段作为管理员功能。

上传表单字段：

- 模型名称
- 版本
- 场景类型：下拉选择，显示 `DetectionScene.display_name`
- 模型类型：下拉选择，决定可上传文件后缀
- 模型文件
- 描述，可选

模型文件后缀约束：

- `yolo11n / yolo11s / yolo11m / yolo11l / yolo11x`：只允许 `.pt`。
- `onnx`：只允许 `.onnx`。

前端必须根据所选 `model_type` 限制可选文件类型，并在选择文件后做本地校验；后端仍必须再次校验，不能只信任前端。

上传流程复用统一的 OSS 文件分片上传模块，与数据集上传使用同一套 OSS 上传会话状态机：

1. 前端选择模型文件。
2. 前端调用后端创建模型上传会话。
3. 后端创建 `model_uploads` 记录，通过 OSS 文件分片上传模块初始化 multipart。
4. 前端按批签发 part URL。
5. 浏览器逐片 PUT 到 OSS。
6. 前端展示上传进度：
   - 已上传分片/总分片
   - 已上传 MB/总 MB
   - 当前速度
   - 预计剩余时间
7. 前端支持暂停/继续上传。
8. 前端支持取消上传，取消时中断当前 XHR 并调用后端 abort。
9. 所有分片完成后调用 complete。
10. 后端校验 OSS 对象并登记 `model_versions + model_artifact_locations`。

上传完成后，模型进入所有模型列表，来源标记为 `上传`。第一阶段上传完成不自动替换当前默认推理模型；默认模型替换统一通过列表中的 `配置` 操作完成。

### 上传控件交互

`模型上传`和`数据集上传`的拖拽上传控件应保持一致：

- 只支持单文件，不支持多选。
- 未选择文件时显示拖拽上传栏。
- 用户拖入或选择文件后，拖拽上传栏替换为单个文件名展示块。
- 文件名展示块显示文件本身名称，例如 `best.pt`、`model.onnx`、`chest_xray_v2.zip`。
- 不显示本地路径，不显示 OSS 路径。
- 文件名较长时允许截断显示。
- 鼠标悬停在文件名上时显示完整文件名。
- 删除已选文件后恢复拖拽上传栏。

### 所有模型管理列表

下半部分用于管理所有模型，样式参考 `数据集管理`页面的表格列表。

筛选区：

- 名称：文本搜索框，适合不定数量值。
- 版本：文本搜索框，适合不定数量值。
- 场景类型：下拉单选框，显示 `DetectionScene.display_name`。
- 模型类型：下拉单选框，例如 `yolo11n / yolo11s / yolo11m / yolo11l / yolo11x / onnx`。
- 来源：下拉单选框，选项为 `全部 / 训练得到 / 上传`。

列表列：

- 模型名称
- 版本
- 场景类型：显示 `DetectionScene.display_name`
- 模型类型：`yolo11n / yolo11s / yolo11m / yolo11l / yolo11x / onnx`
- 来源：
  - 训练来源模型显示训练 ID。
  - 训练 ID 是可点击链接，点击后打开新页面，展示源训练任务的监控页。
  - 上传来源模型显示 `上传`。
- 创建日期
- 操作：
  - 删除
  - 下载权重
  - 配置

`配置`操作弹出配置框。第一阶段配置框只提供“是否替换当前推理默认模型”的确认能力，不包含阈值、类别、权限、说明等其他配置项。

训练 ID 跳转要求：

- 点击训练来源的训练 ID 时，打开新页面。
- 新页面进入 `模型训练`页面，并自动选中该源训练任务。
- `模型训练`页面需要支持从 URL 参数恢复监控上下文，例如 `task_id` 或 `task_uuid`。
- 进入后直接展示该训练任务的监控视图。

## 与现有页面关系

- `模型训练`页面中的“模型版本管理（全局默认模型切换）”应迁移到 `模型管理`页面。
- 迁移完成后，`TrainingPage.vue` 不再提供设置 detection 默认推理模型的操作。
- `TrainingPage.vue` 仍负责训练任务创建、训练任务列表和训练监控。
- `模型管理`页面负责查看当前默认推理模型、上传模型、删除模型、下载权重、配置默认推理模型。
- `检测工作台`选择模型时，应继续只展示 `active` 模型。
- `数据集管理`负责数据集 OSS 生命周期；`模型管理`负责模型 OSS 生命周期，两者不要共用本地目录作为权威数据源。

## 权限设计

第一阶段建议：

- 管理员可查看、上传、删除、设默认、下载全部模型。
- 医生可查看和下载 active 模型，但不能上传、删除或设默认。
- 患者不展示模型管理入口。

如果后续支持用户私有模型，需要在 `model_versions` 或新增关联表中显式记录 `owner_user_id` 和可见范围。

## 兼容与风险

- 当前旧检测服务可能依赖 `model_versions.model_path` 或 `backend/models/best.pt`。实现时需要增加“OSS 权重解析到本地缓存”的逻辑，但权威状态仍以 OSS 为准。
- 删除默认模型必须被阻止，否则检测接口可能找不到可用模型。
- 直接删除 OSS 对象会影响历史检测复现。第一阶段可同步删除，生产阶段更建议先软删除并用 OSS 生命周期策略清理。
- 用户上传模型只做文件级校验不足以保证可推理。后续应增加后台验证任务：加载模型、读取类别、执行最小推理 smoke test，通过后再允许设为默认。

## 实施顺序建议

1. 新增 `model_uploads` 表和后端模型管理路由。
2. 补齐远程训练完成后的 `model_versions + model_artifact_locations` 关联。
3. 新增前端 `ModelPage.vue`、`model.js` API 封装和 `/models` 路由。
4. 将训练页内嵌的默认模型切换能力迁移到 `ModelPage.vue`。
5. 增加 `TrainingPage.vue` URL 参数进入指定训练任务监控的能力，用于模型列表中的训练 ID 跳转。
6. 更新检测服务：默认模型从数据库模型版本解析 OSS 主权重，必要时下载到本地缓存。
7. 统一 `模型上传`和`数据集上传`拖拽控件的单文件展示交互。
8. 增加模型上传 smoke test 和删除/设默认测试。
